# TokenBar — Krill WebView Login + Native Provider (方案 A)

## Goal
Add a dedicated Krill provider to TokenBar with WebView-based login, automatic JWT extraction, and native Krill API data fetching. User installs TokenBar → opens Krill login WebView once → JWT auto-stored → menu bar shows Krill balance/usage forever (auto-refresh until JWT expires, then re-login prompt).

## Current State
- TokenBar has a full WKWebView-based dashboard scraper for OpenAI (NavigationDelegate, JS injection, cookie management in `Sources/CodexBarCore/OpenAIWeb/`)
- Generic `.custom` provider exists with OpenAI-compatible API fetching, but Krill's billing endpoints (`/api/credits`, `/api/subscription`, `/api/request-logs/stats`) are NOT OpenAI-compatible
- Krill API is at `api.krill-ai.com`, uses JWT auth (stored in localStorage as `krill_jwt` after web login)
- Krill API endpoints discovered:
  - `POST /api/auth/login` — email+password login → returns JWT
  - `GET  /api/credits` — wallet balance ($18.35)
  - `GET  /api/subscription` — subscription plans + daily/weekly/monthly quota + request counts
  - `POST /api/request-logs/stats` — total tokens, cache rates, per-channel breakdown, trend
  - `GET  /api/models` — 50+ available models

## Design Decision
Create a **new `.krill` case** in `UsageProvider` enum (not extend `.custom`). Krill has its own API, auth mechanism, and data model — it deserves a first-class provider. This allows:
- Krill-specific branding (🦞 icon, Krill purple color)
- Krill-specific FetchStrategy (JWTAuth with WebView fallback)
- Krill-specific data model (Elite subscription quota, request counts, channel breakdown)
- Clean separation from generic OpenAI-compatible custom providers
